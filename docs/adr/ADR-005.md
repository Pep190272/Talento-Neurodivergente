# ADR-005: Expansion SaaS — 5 Nuevos Bounded Contexts

**Fecha**: 2026-03-09
**Estado**: Aprobado
**Decisor**: Josep + Claude Opus 4.6 (Consultor Tecnico)

## Contexto

Con 4 microservicios operativos (auth, profile, matching, intelligence) y 233 tests pasando, se plantea expandir la plataforma DiversIA para convertirla en un producto SaaS completo con modelo de negocio viable. Las preguntas estrategicas abiertas en NEXT_STEPS.md (modelo de negocio, quien paga) quedan resueltas con esta decision.

## Decision

Definir 5 nuevos Bounded Contexts siguiendo DDD-first (dominio antes que base de datos), cada uno con su schema PostgreSQL aislado. Los schemas comparten la misma instancia PostgreSQL (consistente con ADR-004).

## Nuevos Bounded Contexts

| Schema | Servicio Futuro | Puerto | Responsabilidad |
|--------|----------------|--------|-----------------|
| `subscriptions` | subscription-service | 8005 | Planes, suscripciones, Stripe, facturacion |
| `learning` | learning-service | 8006 | Cursos adaptativos, modulos, certificaciones |
| `community` | community-service | 8007 | Grupos, posts, eventos, moderacion |
| `marketplace` | marketplace-service | 8008 | Proveedores, servicios, reservas, comisiones |
| `analytics` | analytics-service | 8009 | Metricas DEI, reportes ESG/CSRD, usage events |

## Diseno DDD-first

Cada bounded context se diseno siguiendo este proceso:

1. **Identificar Aggregates** con sus invariantes de dominio
2. **Definir Value Objects** inmutables con validacion en construccion
3. **Modelar comportamiento** rico en las entidades (no anemico)
4. **Derivar tablas** como reflejo del modelo, no al reves

### Aggregates por contexto

| Contexto | Aggregates |
|----------|-----------|
| Subscriptions | Plan, Subscription, Invoice |
| Learning | Course (con Module, Lesson), Enrollment, Certificate |
| Community | Group, Post, Comment, Membership, Event |
| Marketplace | Provider, Service, Booking, Review |
| Analytics | MetricSnapshot, DeiReport, UsageEvent |

## Convenciones SQL (consistentes con v1)

- `snake_case` columnas, tablas plural minusculas
- `VARCHAR(25)` para IDs (CUID)
- `TIMESTAMP WITH TIME ZONE` para timestamps
- `JSONB` para estructuras anidadas complejas
- `TEXT[]` para arrays de strings
- Enums con prefijo de schema (`subscriptions.plan_tier`)
- No FK cross-schema (aislamiento de bounded context)
- Indices en: status, foreign keys, timestamps para range queries

## Alternativas Consideradas

| Alternativa | Pros | Contras | Veredicto |
|------------|------|---------|-----------|
| Anadir tablas a schemas existentes | Rapido | Rompe bounded contexts, acoplamiento | Rechazado |
| Base de datos separada por servicio | Maximo aislamiento | Complejidad operacional, un solo VPS | Rechazado |
| NoSQL (MongoDB) para analytics | Flexible | Stack adicional, mas complejidad | Rechazado |
| Solo 2-3 contextos iniciales | Menos trabajo | Diseno incompleto, refactor posterior | Rechazado |

## Consecuencias

### Positivas
- Modelo de negocio definido y viable (breakeven ~mes 7)
- Cada contexto puede implementarse independientemente
- Escalabilidad: nuevos servicios sin tocar los existentes
- Dominio modelado antes que infraestructura (DDD correcto)

### Negativas
- 9 schemas totales (complejidad operacional)
- Mas migraciones SQL que mantener
- Latencia inter-servicio creciente con mas servicios

### Riesgos
- **Mitigado**: Schemas en misma PostgreSQL (latencia 0 para lecturas)
- **Mitigado**: HTTP sincrono inicialmente (consistente con ADR-004 Decision 1)
- **Aceptado**: 5 servicios nuevos son mucho trabajo; implementar en orden de prioridad
- **Futuro**: Event bus cuando > 100 usuarios concurrentes

## Prioridad de Implementacion

```
Fase 1 (Mes 1): subscriptions — Stripe + planes → monetizacion
Fase 2 (Mes 2): analytics — metricas + DEI reports → valor para empresas
Fase 3 (Mes 3): learning — cursos adaptativos → retention candidatos
Fase 4 (Mes 4): community — grupos + eventos → engagement
Fase 5 (Mes 5): marketplace — proveedores → ecosistema completo
```
