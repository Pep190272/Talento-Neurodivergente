# ADR-004: Microservicios en vez de Monolito Modular

**Fecha**: 2026-03-04
**Estado**: Aprobado
**Decisor**: Josep + Atlas (GACE)

## Contexto

Al migrar a Python/FastAPI, se evaluó si mantener un monolito (modular) o implementar microservicios desde el inicio. El ADR-002 (Feb 2026) recomendaba monolito para equipo pequeño, pero las circunstancias han cambiado con el rewrite completo.

## Decisión

Implementar 4 microservicios con Clean Architecture interna, comunicación REST síncrona, y base de datos compartida con schemas separados.

## Servicios Definidos

| Servicio | Puerto | Responsabilidad | Issues |
|----------|--------|----------------|--------|
| auth-service | 8001 | Identidad, JWT, registro | #38, #41, #69 |
| profile-service | 8002 | Perfiles, quiz, games, frontend | #42-#47 |
| matching-service | 8003 | Matching trilateral 24D | #40, #81, #83 |
| intelligence-service | 8004 | LLM, análisis, transparencia IA | #60, #84, #85, #88 |

## Razones

1. **Bounded contexts claros**: Auth, Profiles, Matching, AI son dominios independientes
2. **Escalabilidad**: matching-service consume más CPU → escalar independientemente
3. **Aprendizaje**: Microservicios es una competencia profesional valiosa para el CV
4. **Alineación con negocio**: Cada servicio podría ser un producto independiente
5. **Deploy independiente**: Actualizar matching sin tocar auth
6. **Infraestructura existente**: Dokploy + Docker Compose ya están configurados

## Alternativas Consideradas

| Alternativa | Pros | Contras | Veredicto |
|------------|------|---------|-----------|
| Monolito modular | Simple, rápido | No aprende microservicios, menos CV | Rechazado |
| Monolito → split posterior | Menos riesgo | Boundaries se difuminan con el tiempo | Rechazado |
| 6+ microservicios | Máxima separación | Demasiada complejidad operacional | Rechazado |

## Pragmatismo

Para mantener la complejidad manejable:

1. **Shared DB**: Base de datos compartida con schemas separados (no DB por servicio)
2. **REST síncrono**: No message queues ni eventos async (por ahora)
3. **JWT compartido**: auth-service firma, los demás verifican con misma clave
4. **Docker Compose**: No Kubernetes (innecesario para esta escala)
5. **Shared kernel**: Librería Python con tipos comunes (BaseEntity, NeuroVector24D, JWT utils)

## Consecuencias

### Positivas
- Separación real de concerns
- Deploy independiente por servicio
- Testing aislado por servicio
- Aprendizaje de patrones distribuidos
- OpenAPI por servicio (documentación automática)

### Negativas
- Latencia inter-servicio (HTTP calls entre servicios)
- Complejidad operacional (4 Dockerfiles, 4 Alembics)
- Debugging más difícil (tracing distribuido necesario después)
- Duplicación temporal de código compartido

### Riesgos
- **Mitigado**: Shared kernel para evitar duplicación
- **Mitigado**: Docker Compose simplifica orquestación local
- **Aceptado**: Latencia inter-servicio aceptable para MVP (<100ms en red local)
- **Futuro**: Redis Streams o RabbitMQ si se necesita comunicación async

## Métricas de Éxito
- [ ] 4 servicios levantando con `docker-compose up`
- [ ] Health checks respondiendo en todos
- [ ] Tests pasando independientemente por servicio
- [ ] Latencia inter-servicio < 100ms
- [ ] Zero credenciales en texto plano
