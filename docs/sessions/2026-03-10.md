## Contexto

Primera sesion de deploy real a produccion. Todos los servicios estaban listos como codigo con tests, pero nunca habian corrido juntos en Docker. Objetivo: desplegar los 4 microservicios + PostgreSQL + nginx gateway en el VPS Hostinger via Dokploy, accesible en `app.diversia.click`.

## Trabajos Realizados

### 1. Docker Compose produccion para Dokploy

Creacion de `docker-compose.prod.yml` separado del de desarrollo, con:
- PostgreSQL 16 con health checks
- Ollama con limites de memoria (6GB)
- 4 microservicios con Dockerfiles multi-stage
- nginx gateway como punto de entrada
- Variables de entorno con defaults seguros

### 2. Resolucion de problemas de build (cadena de 6 fixes)

Cada servicio necesito correcciones iterativas al pasar de desarrollo local a contenedores Docker:

| Fix | Problema | Solucion |
|-----|----------|----------|
| 1 | `init-schemas.sql` fallaba con credenciales especiales | URL-encode de password + GRANT dinamico |
| 2 | `psycopg2` no incluido en Dockerfiles | Agregado `psycopg2-binary` como dependencia |
| 3 | PYTHONPATH no apuntaba a `/app` en contenedores | Agregado `ENV PYTHONPATH=/app` en Dockerfiles |
| 4 | Variables de entorno sin defaults en compose | Defaults con `${VAR:-default}` + start_period 30s |
| 5 | Alembic migracion no era idempotente | Refactorizado para sobrevivir reintentos |
| 6 | `email-validator` no instalado | Agregado como dependencia de auth-service |

### 3. Resolucion de problemas de red y routing (cadena de 6 fixes)

Dokploy usa Traefik como reverse proxy. Conectar nginx gateway con Traefik requirio multiples iteraciones:

| Fix | Problema | Solucion |
|-----|----------|----------|
| 1 | Traefik labels manuales conflictaban con Dokploy | Eliminados — Dokploy gestiona Traefik |
| 2 | Gateway no accesible desde Traefik | Conectar gateway a `dokploy-network` |
| 3 | CSP bloqueaba Alpine.js `x-data` | Agregado `unsafe-eval` a Content-Security-Policy |
| 4 | Red `default` eliminada dejaba servicios sin red | Reconectar todos los servicios a `dokploy-network` |
| 5 | `dokploy-network` en todos los servicios — innecesario | Solo gateway necesita `dokploy-network` |
| 6 | nginx cacheaba DNS al arrancar → 502 Bad Gateway | Red `internal` explicita + `resolver 127.0.0.11` con variables `set $upstream` |

### 4. Arquitectura de red final (produccion)

```
Internet → Traefik (Dokploy) → dokploy-network → nginx gateway (:8000)
                                                        |
                                                   internal network
                                                        |
                            +----------+----------+----------+----------+
                            |          |          |          |          |
                          auth     profile    matching  intelligence    db
                         :8001     :8002      :8003      :8004       :5432
```

- **Red `internal`** (bridge): comunicacion entre todos los servicios
- **Red `dokploy-network`** (external): solo gateway, para que Traefik lo alcance
- **DNS dinamico**: nginx usa `resolver 127.0.0.11 valid=10s` + variables `set $upstream` para re-resolver DNS en cada peticion

### 5. Estado final: app.diversia.click OPERATIVA

- 4 microservicios corriendo y healthy
- nginx gateway ruteando correctamente
- Health checks respondiendo 200 OK
- Registro de usuarios funcional
- Landing page, login, dashboard accesibles

## Lecciones Aprendidas

1. **Docker Compose `default` network** no se comporta de forma predecible cuando se mezcla con redes externas — usar siempre redes nombradas explicitas
2. **nginx upstream blocks** cachean DNS al arrancar y no lo re-resuelven — usar `resolver` + variables para entornos Docker donde los IPs cambian
3. **Dokploy gestiona Traefik** automaticamente — no poner labels manuales
4. **Las migraciones deben ser idempotentes** — en contenedores se reintentan al reiniciar
5. **12 commits iterativos** fueron necesarios para pasar de "funciona local" a "funciona en produccion" — normal en primer deploy de microservicios
